## TL;DR

一个本地运行、跨平台（Mac + Windows）的小工具，把 PDF 文件转成 Markdown。默认严格本地处理（文件不出电脑），用户可选择性触发基于 Claude 的日→中翻译，一次生成双语两份 MD 文件。MVP 基于 Gradio + MinerU，目标是替代「命令行跑 markitdown + 手动复制到 Claude 翻译」的现有工作流，把一份 PDF 的端到端处理时间从 10+ 分钟压到 2 分钟以内。

---

## Problem Statement

日常工作中要频繁处理日文 PDF（老板发来的商务提案、内部资料等）。当前处理这类文档的流程低效且易出错：

- **markitdown 是命令行工具**，Windows 下每次用都要开终端、记参数，体验糟糕；并且对复杂排版、扫描件处理能力不足。
- **转换完成后**还要手动复制到 Claude 做翻译，步骤多、容易在粘贴中丢格式。
- **每一份文件都是重复劳动**——拖、转、复制、粘贴、等、保存——每份约 10 分钟机械操作，且容易分心出错。
- **没有隐私控制**——要么全手动保证不上云，要么全自动牺牲隐私，没有中间态。

这让文档处理成为日常工作里持续性的摩擦。

---

## Goals

1. **把单份「PDF → 可读的中文 Markdown」时间从 10+ 分钟压到 <2 分钟**（含人工介入）
2. **消除命令行**：图形化界面，拖拽即用，Windows + Mac 双平台
3. **给用户明确的隐私控制**：默认不外传，仅在用户显式授权时才调用 Claude
4. **在老板实际发来的文档（AI 生成 / Office 导出的 native-text PDF）上，转换质量达到「基本不需要手动修」**
5. **作为 PM 实践项目**，完整跑一遍「定义问题 → 定范围 → 立假设 → 验证 → 发布 → 复盘」的流程

---

## Non-Goals

| 明确不做 | 理由 |
|---|---|
| 批量处理（一次 >2 份文件） | 实际使用场景几乎都是 1-2 份的 ad-hoc 处理；批量会大幅增加 UI 和错误处理复杂度，收益不成正比 |
| 支持 Excel / Word / PPT / HTML 等其他格式 | 实际场景 90%+ 是 PDF；MVP 先聚焦做好最难最重要的 PDF |
| 支持扫描件、竖排、试卷等复杂版面 PDF | MinerU 在这些场景已验证质量不够（见「风险」），需要完全不同的技术栈，超出 MVP 范围 |
| 内置翻译以外的 AI 能力（摘要、提炼、结构化抽取） | 这些工作用户已经直接用 Claude 做，工具不应越俎代庖 |
| 做成 Electron / Tauri 原生桌面应用 | Gradio 已满足跨平台 + UI 需求；打包复杂度 vs 价值不成正比 |
| 集成 OCR（处理扫描件） | 和「支持扫描件」同理，非 MVP 要解决的问题 |
| 追求丰富的 UI / 视觉设计 | 单用户个人工具，功能优先；过度设计浪费时间 |

## User Stories

### 核心流程

1. **As a** 用户，**I want** 把 PDF 拖入工具界面，**so that** 我不需要打开终端或记命令
2. **As a** 用户，**I want** 在界面里看到转换后的日文 Markdown，**so that** 我可以判断转换质量并决定是否继续
3. **As a** 用户，**I want** 一键把日文 Markdown 翻译成中文，**so that** 我不需要复制粘贴到 Claude
4. **As a** 用户，**I want** 工具把日文 MD 和中文 MD 都保存成文件，**so that** 我可以以后查阅原文和译文

### 隐私控制

5. **As a** 用户，**I want** 默认情况下文件处理完全在本地完成，**so that** 敏感文件不会被误传到外部
6. **As a** 用户，**I want** 只有在我显式点击「翻译」时才调用 Claude，**so that** 我对每份文件独立决定隐私取舍
7. **As a** 用户，**I want** 第一次点翻译时看到数据外传的明确提示，**so that** 我不会无意识地把敏感内容发出去

### 失败处理

8. **As a** 用户，**I want** 转换或翻译失败时看到清晰错误信息，**so that** 我知道是 PDF 问题还是工具问题
9. **As a** 用户，**I want** 翻译失败时仍然保留日文 MD，**so that** 已经完成的转换劳动不白费

## Requirements

### P0 — Must Have (MVP 必备)

**PDF 转 Markdown**
- [ ] 支持拖拽或点击选择上传单个 PDF 文件
- [ ] 使用 MinerU 作为 PDF 解析引擎
- [ ] 在界面内显示转换后的日文 Markdown（预览 + 可复制）
- [ ] 把日文 Markdown 保存为文件（文件名格式：`{原文件名}.ja.md`）
- [ ] 转换失败时显示具体错误信息，不静默崩溃

**可选翻译**
- [ ] 在日文 MD 预览下方提供「翻译为中文」按钮
- [ ] 点击后调用 Claude，将日文 MD 翻译成中文 MD
- [ ] 翻译完成后在界面显示中文 MD 预览
- [ ] 把中文 MD 保存为文件（`{原文件名}.zh.md`）
- [ ] 未点击按钮时，工具不发起任何外部网络请求

**隐私控制**
- [ ] 不上传任何遥测 / 日志 / 崩溃数据
- [ ] 第一次点击「翻译」时弹出确认对话框：「将通过 Claude 翻译，内容会发送到 Anthropic 服务器。继续？」，含「本次会话不再提醒」选项
- [ ] 对外网络调用的路径可用网络监控工具清晰观察（便于未来审计）

**跨平台**
- [ ] Windows 10/11 上能运行
- [ ] macOS (Intel + Apple Silicon) 上能运行
- [ ] 启动方式一致（`python app.py` 或包装的启动脚本）

### P1 — Should Have (快速跟进)

- [ ] 日文 MD 预览支持在界面内编辑（用户可在翻译前手动修正转换错误）
- [ ] 长文档分块翻译（自动分段调用 Claude + 保持 Markdown 结构跨块一致）
- [ ] 输出目录可配置（默认：PDF 源文件同目录）
- [ ] 翻译时显示进度指示
- [ ] 支持同时上传 2 份文件并行处理

### P2 — Future Consideration (不做但架构要允许)

- **批量模式**（针对将来可能的场景 B）：核心转换模块要可脚本化调用，**不能和 Gradio UI 耦合**
- **其他输入格式**（Excel / Word / PPT）：引擎抽象成「格式 → MD」pipeline，加格式只需要加 adapter
- **本地 LLM 翻译**（替代 Claude）：翻译 provider 做成可替换接口（如 `class Translator` 抽象）
- **Windows Explorer 右键菜单集成 / macOS 服务菜单集成**

## User Flow (MVP)

```
1. 用户双击启动脚本 / 终端运行 python app.py
2. 浏览器自动打开 localhost 页面
3. 用户拖入 PDF → 文件上传到本地临时目录
4. 工具自动开始转换（spinner + "正在处理..."）
5. 转换完成 → 显示日文 MD 预览 + 自动保存 {name}.ja.md
6. 用户决策分支：
   a. 满意 → 关掉页面，或继续拖下一份
   b. 不满意 → 手动在预览框里修（P1）
   c. 需要中文 → 点击「翻译为中文」按钮
7. 如果点翻译：
   - 首次：弹出隐私提示，用户确认
   - 工具调用 Claude（显示进度）
   - 翻译完成 → 显示中文 MD 预览 + 自动保存 {name}.zh.md
8. 用户可复制任一预览内容，或在文件系统找生成的 .md 文件
```

## Technical Approach

**Stack**
- Python 3.11+
- Gradio (UI + localhost 托管)
- MinerU (PDF → MD 解析引擎)
- Anthropic SDK 或 `claude` CLI 调用（翻译）

**关键架构决策**
- **核心转换逻辑封装成独立模块**，不依赖 Gradio，为将来批量化 / 其他 UI 留余地
- **翻译通过接口抽象**（`class Translator` → 默认 `ClaudeTranslator`），未来可替换为 `OllamaTranslator`
- **所有文件 I/O 和网络调用保持清晰边界**，便于隐私审计
- **配置通过一个简单 YAML / TOML 文件**，不引入复杂配置系统

**分发策略**
- MVP 阶段不打包 installer，保持 Python script 形态
- 仓库提供 `run.bat` (Windows) 和 `run.sh` (Mac) 启动脚本
- P1 或之后：考虑用 PyInstaller / Briefcase 打包成单文件可执行

## Success Metrics

因为是**个人工具**，指标以自用体验为主。不追求 DAU / MAU 这类产品指标。

### Leading Indicators (上线 2 周内验证)

| 指标 | 目标 | 衡量方式 |
|---|---|---|
| 采纳率 | 下一份日文 PDF 默认用本工具 > 80% | 自己记录 |
| 首次成功率 | 转换「一次过关，不需手动修」的比例 > 70% | 每次处理后自评 |
| 端到端耗时 | 拖入 PDF 到拿到中文 MD < 2 分钟 | 自己掐表 |
| 崩溃率 | 使用中崩溃 / 卡死比例 < 5% | 自己记录 |

### Lagging Indicators (上线 4-8 周验证)

| 指标 | 目标 |
|---|---|
| 真实工作节省 | 每周花在文档处理上的时间下降 > 50% |
| 是否成为日常标配 | 处理日文 PDF 时「默认选择」已经是本工具 |
| 隐私事故 | 期间没有违反公司信息安全政策的事件 |

**衡量方式**：每周花 5 分钟做一次迷你复盘，记在个人日志里。

## Risks & Assumptions

| 风险 / 假设 | 严重度 | 应对 |
|---|---|---|
| MinerU 对未来多样化的老板 PDF 质量稳定 | 高 | 已用 1 份样本验证；后续多样化样本持续观察；如果某类 PDF 持续翻车，明确写入已知限制 |
| Claude API / Claude Code 调用额度够用 | 中 | 个人用量通常不是问题；加重试 + 超限时明确提示 |
| Gradio 在 Windows 和 Mac 的 UX 差异 | 低 | 浏览器 UI 差异较小；Phase 2 双平台测试能发现 |
| 业余时间能否持续迭代 | 中 | Scope 已砍到足够小；MVP 一周可交付；之后按需 |
| 公司政策变化导致不能用 Claude 翻译 | 中 | 架构已为本地 LLM 留接口，切换成本可控 |
| 自己会过度工程化（陷入「做 app」的完美主义） | 中 | 用本文档作为锚，每次想加功能先对照 Non-Goals |

## PM 自我复盘备注（仅供自己用）

**做这份 spec 学到了什么**

- Brainstorm 一上来「我要做个工具」是**方案**不是问题，真正的问题要反复往下挖
- 用户（即我自己）对约束的描述往往自相矛盾——「文件不出本地」和「用 Claude 翻译」本质冲突，要显式解决
- 真实的质量验证（30 分钟 MinerU 测试）胜过一千字的脑补，**比写文档更重要的是先做关键实验**
- 「opt-in privacy」这种把选择权交给用户的设计，比 PM 替用户做决定更成熟
- Non-Goals 这一节帮自己挡住了至少 3 次想加功能的冲动

**下一次做 spec 要更好的点**

- 老板的 PDF 具体样本多样性还不够，应该在写 spec 前多攒几份测试样本
- Success metrics 还可以更具体（比如「每周节省时间 > 50%」这个目标有点模糊，应该定义「从 X 小时降到 Y 小时」）
